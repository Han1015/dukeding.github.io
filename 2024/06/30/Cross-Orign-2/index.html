CORS(Cross-Origin Resource Sharing)

Many people might confuse CORS (Cross-Origin Resource Sharing) as a security protocol, but it’s not. CORS is a system composed of a series of HTTP headers that determine whether the client browser should block frontend JavaScript code from accessing responsesto cross-origin requests. CORS can be explained as a mechanism that provides a secure way for the Same-Origin Policy (SOP) to allow cross-origin requests under specific conditions defined by the server.

Request Headers

  • Origin: Indicates the origin (protocol, host, and port) of the request.

Response Headers

  • Access-Control-Allow-Origin: Specifies which origin(s) are allowed (e.g., *, specific domain).
  • Access-Control-Allow-Methods: Lists permitted HTTP methods (e.g., GET, POST).
  • Access-Control-Allow-Headers: Specifies allowed custom request headers.
  • Access-Control-Expose-Headers: Lists response headers accessible to the client.
  • Access-Control-Allow-Credentials: Indicates if credentials (cookies, HTTP auth) are allowed (true or false).
  • Access-Control-Max-Age: Specifies how long preflight response can be cached (in seconds).

Preflight Requests

For complex requests, the browser sends an OPTIONS request with the following headers:

  • Access-Control-Request-Method: The intended HTTP method.
  • Access-Control-Request-Headers: Custom headers the client wants to send.

The process of cross domain requests

Simple Requests

A request is considered simple if it meets all the following conditions:

  1. HTTP Methods: The request uses one of the following methods:

    • GET
    • HEAD
    • POST

  2. Headers: The request contains only headers that are automatically set by the browser, such as:

    • Accept
    • Accept-Language
    • Content-Language
    • Content-Type (with values application/x-www-form-urlencoded, multipart/form-data, or text/plain)

  3. No Custom Headers: The request does not include any custom headers.

Behavior: If your request is defined as a simple request by the browser, you will not send a preflight request for this request, so the request will be successfully sent and arrive at the server. The service will make its own judgment and respond. If the response header does not contain Access Control Allow Origin or the attribute value does not contain the current domain name, it will be blocked from obtaining the response content.

Non-Simple Requests

Requests that do not meet the criteria for simple requests are classified as non-simple. This includes requests that:

  • Use HTTP methods other than GET, HEAD, or POST.
  • Include custom headers not automatically set by the browser.
  • Use a Content-Type other than application/x-www-form-urlencoded, multipart/form-data, or text/plain.

Behavior: For non-simple requests, the browser performs a preflight request using the OPTIONS method to determine if the actual request is safe to send. The OPTIONSmethod will include the Access-Control-Request-Method in the request header to inform the server of the actual request method, and the Access-Control-Request-Headers to inform the server of the custom header fields included in the actual request. The server will then decide whether the actual request is allowed.

And the real response needs to contain the correct access-control-allow-origin, the response can be received. If the access-control-allow-origin is missing in the response to an actual CORS request, the browser will block the response, and the requesting web application will not be able to access the content.
